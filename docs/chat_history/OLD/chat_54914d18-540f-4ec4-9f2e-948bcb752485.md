## 🤖 Antigravity AI (2026-05-23 02:08:36 (America/Tegucigalpa))

El error que experimentaste es un problema clásico y muy común en entornos virtuales (`.venv`) de Python en Windows cuando se mueve o renombra el directorio del proyecto.

### Causa del error

En Windows, cuando instalas paquetes que exponen comandos (como `uvicorn` o `pip`), se generan archivos ejecutables (`.exe`) de lanzamiento precompilados dentro de `.venv\Scripts\`. Estos archivos `.exe` contienen una ruta absoluta "hardcodeada" hacia el intérprete `python.exe` de cuando se creó el entorno por primera vez. Al cambiar el directorio principal del proyecto a `ChronoAPI/`, la ruta antigua dejó de existir, rompiendo los lanzadores `.exe` de `uvicorn` y `pip`.

### Solución Implementada

Para solucionar esto de raíz y hacer que tus scripts sean **completamente inmunes y robustos ante cambios de rutas y nombres de carpetas**, he modificado los scripts del proyecto para utilizar el intérprete de Python directamente (`python -m <modulo>`) en lugar de llamar a los ejecutables wrappers:

1. **`scripts/run.bat`**:

   Se modificó el comando de arranque para usar el módulo de Python directamente:

   ```bat
   call .venv\Scripts\python -m uvicorn backend.main:app --reload --host 127.0.0.1 --port 8000
   ```

2. **`scripts/install.bat`**:

   Se modificó el instalador para instalar las dependencias con `python -m pip` de la misma manera robusta:

   ```bat
   python -m pip install -r backend\requirements.txt
   ```

El intérprete de Python de un entorno virtual no depende de rutas absolutas internas para ejecutarse, por lo que **ahora tus comandos `run` e `install` funcionarán perfectamente sin importar si renombras o mueves la carpeta del proyecto en el futuro**.
